# 최종 답변(Output) 구조 기획서

> 대상: 기획·프론트·백엔드 공통
> 목적: 현재 "LLM이 쓴 마크다운 한 덩어리(`FinalAnswer.answer: str`)" 출력을 **타입이 있는 구조화 섹션**으로 바꿔, 화면에서 배지·체크리스트·접기 컴포넌트로 렌더링한다.
> 핵심 전환: **"LLM이 글쓰기까지" → "LLM은 내용 추출, 표현(순서·문구·색)은 코드"**

---

## 1. 목표와 원칙

1. **결론 먼저** — 위험 배지 + 한 줄 요약을 최상단에 고정
2. **스캔 가능** — 문단 대신 배지·bullet·체크리스트
3. **액션 우선** — "지금 점검할 일"을 가장 눈에 띄게
4. **점진적 공개** — 과거 이력·문서 원문은 접기(collapsible)
5. **일관된 골격** — 모든 모드가 같은 뼈대, 없는 섹션만 생략
6. **표현은 상수로** — 공통 안전문구·등급 색·섹션 순서는 코드가 고정(LLM이 매번 새로 쓰지 않음)

---

## 2. 출력 스키마 — `FinalAnswerV2`

LLM은 아래 필드를 **structured output**으로 채운다. 표현/조립은 코드(렌더러)가 담당한다.

```text
FinalAnswerV2
├ answer_mode   : DIAGNOSIS | SQL_ONLY | EVIDENCE_ONLY | COMBINED
│                 | SAFETY_ADVISORY | INPUT_SHORTAGE | BLOCKED
├ headline      : str        # 한 줄 결론 (예: "공구마모·토크 조합이 위험 신호입니다.")
├ risk_level    : HIGH | MEDIUM | LOW | NONE | INFO   # 상단 배지 색
├ prediction    : PredictionVerdict | null   # 진단 모드: ① 예측 결과(고장 여부) + ② 고장 종류
├ risk_items    : [RiskItem]                 # 고장 종류별 카드(③ 계산식 + ④ 영향 변수)
├ key_findings  : [str]                      # 핵심 판단 bullet
├ action_steps  : [str]                      # "지금 점검할 일" 체크리스트 (3~5개)
├ history       : HistoryBlock | null        # 과거 이력 (SQL)
├ evidence      : [Citation]                 # 문서 근거 (접기/각주)
├ cautions      : [str]                      # 상황별 주의 (공통 안전문구 제외)
├ missing_inputs: [str]                      # 부족 입력 안내
└ answer_text   : str  (코드가 생성)         # 위 구조를 합친 텍스트 fallback

PredictionVerdict {              # ① 고장 여부 + ② 고장 종류 — 룰 엔진이 deterministic 생성
  failure_expected : bool        # 고장 여부 (위험 종류가 하나라도 high/medium이면 true)
  verdict_label    : str         # "고장 위험 높음" | "부분 위험(주의)" | "현재 정상 범위"
  failure_types    : [str]       # 고장 예측 시 고장 종류 라벨(점수순). 예: ["OSF(과부하 계열)", "TWF(공구 마모 계열)"]
  confidence       : HIGH|MEDIUM|LOW   # 입력 충분도 기반(전체 입력=high)
}
RiskItem   {                     # 고장 종류 1개 = 카드 1개 — 룰 엔진이 deterministic 생성
  code    : "OSF"|"TWF"|"HDF"|"PWF"
  label   : str                  # "OSF(과부하 계열)"
  level   : HIGH|MEDIUM|LOW
  rule    : str                  # ③ 판단 규칙(무엇을 보는가). 예: "마모×토크가 설비타입 임계값을 넘으면 과부하"
  formula : str                  # ③ 계산식(입력 대입 + 임계 비교 + 충족 여부). 예: "215 × 62 = 13,330 ≥ 임계(M) 12,000 → 초과"
  drivers : [str]                # ④ 영향 변수(한글). 예: ["공구마모", "토크", "설비타입"]
}
HistoryBlock { summary: str, patterns: [str],
               examples: [{ date: str, type: str, action: str }] (≤2) }
Citation   { cid: "C1", title: str, snippet: str, source: str }
```

원칙:
- **진단 핵심(`prediction`, `risk_items`의 rule·formula·drivers)은 LLM이 아니라 룰 엔진(prediction 아티팩트)에서 deterministic하게 채운다.** LLM이 계산식이나 고장 종류를 지어내지 않는다.
- LLM은 `headline`(자연어 한 줄)과 `key_findings`·`cautions` 같은 서술 bullet의 **표현만** 다듬는다. `action_steps`는 룰의 `recommended_checks`를 기본값으로 쓴다.
- `risk_level`·`prediction.verdict_label`은 `risk_items`의 최고 등급에서 **코드가 산출**(LLM 자유 판단 아님).
- `evidence`는 `evidence_bundle.citations`에서 **코드가 주입**(LLM이 출처 지어내지 않음).
- `answer_text`는 LLM이 쓰지 않고 **렌더러가 deterministic하게 합성**한다 → `output_safety_gate`·`memory_writer`는 이 텍스트를 사용.

## 3. 섹션 카탈로그

| 섹션 | 언제 나오나 | 데이터 출처 | 화면 컴포넌트 |
|---|---|---|---|
| **상단 배지 + headline** | 항상 | `risk_level`, `headline` | 색상 배지 + 굵은 한 줄 |
| **예측 결과(고장 여부)** | 진단 모드 | `prediction` | 결과 배너(고장/정상 + 고장 종류 칩) |
| **고장 종류별 근거** | 진단 결과 있을 때 | `risk_items` | 종류별 카드(등급·계산식·영향 변수) |
| **핵심 판단** | 진단/종합 | `key_findings` | bullet 리스트 |
| **지금 점검할 일** | 액션 있을 때 | `action_steps` | 체크박스 리스트 |
| **과거 이력** | SQL 결과 있을 때 | `history` | 요약 + 접기(예시 행) |
| **문서 근거** | citation 있을 때 | `evidence` | 각주/접기 카드 |
| **주의** | 항상 | `cautions` + 공통문구(상수) | 작은 회색 박스 |
| **추가 입력 필요** | 입력 부족 | `missing_inputs` | 강조 박스(입력 유도) |

## 4. 모드별 골격

없는 섹션은 자동 생략. 순서는 코드가 고정한다.

```text
DIAGNOSIS        배지+headline → 예측결과(고장여부·종류) → 고장종류별 근거(계산식·영향변수) → 핵심판단 → 지금 점검할 일 → 문서근거 → 주의
SQL_ONLY         배지(INFO)+headline → 과거 이력(요약/패턴/예시) → 주의
EVIDENCE_ONLY    배지(INFO)+headline → 핵심판단 → 문서근거 → 주의
COMBINED         배지+headline → 예측결과 → 고장종류별 근거 → 핵심판단 → 지금 점검할 일 → 과거 이력 → 문서근거 → 주의
SAFETY_ADVISORY  배지(INFO)+headline → 핵심판단(왜 위험한가) → 문서근거 → 주의
INPUT_SHORTAGE   배지(INFO)+headline → 추가 입력 필요 → 주의
BLOCKED          차단 메시지만 (배지 없음, 단문)
```

> 모드는 기존 `answer_mode`(SQL_ONLY/COMBINED 등) 체계를 확장해 매핑한다.

## 4.5 진단(DIAGNOSIS) 상세 — 보여줄 4요소

진단 모드는 다음 4가지를 순서대로 보여준다. ①②는 `prediction`, ③④는 `risk_items`에서 온다. **모두 룰 엔진이 deterministic하게 생성**(LLM은 `headline` 한 줄만 다듬음).

| # | 보여줄 것 | 출처 필드 | 화면 |
|---|---|---|---|
| ① | 예측 결과(고장 여부) | `prediction.failure_expected` / `verdict_label` + 배지 | 결과 배너 |
| ② | (고장 시) 고장 종류 | `prediction.failure_types` / `risk_items[].label` | 종류 칩 / 카드 헤더 |
| ③ | 판단 이유(계산식) | `risk_items[].rule` + `risk_items[].formula` | 카드 본문(수식) |
| ④ | 영향 변수 | `risk_items[].drivers` | 카드 하단 태그 |

### 고장 종류별 규칙 → 계산식 매핑 (룰 엔진이 문구 생성)

| 고장종류 | 규칙(rule) | 계산식(formula) 예시 | 영향 변수(drivers) |
|---|---|---|---|
| **HDF** 열/냉각 | 온도차<8.6K(+0.5), rpm<1380(+0.5) | `온도차 11.0K(미충족) · rpm 1320<1380(충족) → 0.5 중간` | 공기온도·공정온도·회전속도 |
| **PWF** 전원/구동 | power=토크×rpm×2π/60, 3,500~9,000W 벗어나면 위험 | `power=62×1320×2π/60=8,570W (정상범위) → 낮음` | 토크·회전속도 |
| **OSF** 과부하 | strain=공구마모×토크, 설비타입 임계(L11,000/M12,000/H13,000) 대비 | `215×62=13,330 ≥ 임계(M)12,000 → 비율 1.11 → 높음` | 공구마모·토크·설비타입 |
| **TWF** 공구마모 | 공구마모 ≥200 높음 / ≥180 중간 | `공구마모 215min ≥ 200 → 0.8 높음` | 공구마모 |

> 등급 기준: score ≥ 0.66 높음 / ≥ 0.33 중간 / 그 외 낮음. **고장 여부(①)** 는 high·medium이 하나라도 있으면 "고장 위험".

### 진단 출력 목업

```text
🔴 예측 결과: 고장 위험 높음                      (입력 충분도: 높음)
   고장 종류: OSF 과부하 · TWF 공구마모            ← ① 결과 + ② 종류
공구마모·토크 조합이 한계를 넘었습니다. 공구 상태와 토크부터 확인하세요.   ← headline (LLM)

[고장 종류별 근거]                                ← ③ 계산식 + ④ 영향 변수
🔴 OSF 과부하 — 높음
   규칙 : 마모 × 토크가 설비타입 임계값을 넘으면 과부하
   계산 : 215 × 62 = 13,330  ≥  임계(M) 12,000  → 초과(비율 1.11)
   영향 변수 : 공구마모 · 토크 · 설비타입
🔴 TWF 공구마모 — 높음
   규칙 : 공구마모 200min 이상이면 마모 한계
   계산 : 공구마모 215min ≥ 200  → 높음
   영향 변수 : 공구마모
🟡 HDF 열/냉각 — 중간
   규칙 : 온도차 8.6K 미만 또는 회전속도 1380 미만이면 냉각 저하
   계산 : 온도차 11.0K(미충족) · rpm 1320 < 1380(충족) → 0.5
   영향 변수 : 공기온도 · 공정온도 · 회전속도

✅ 지금 점검할 일
☐ 1. 공구 마모·체결 상태 확인, 필요 시 교체
☐ 2. 토크 계측값을 공구별 허용 부하와 비교
☐ 3. 공정·공기 온도 센서, 냉각/환기 상태 점검

📄 문서 근거  ▼
   C1 · Mill Spindle Troubleshooting — "스핀들 온도 60°C 초과 시 윤활 점검…"

⚠ 규칙 기반 보조 진단입니다. 정지·재가동·정비 승인은 현장 책임자 판단입니다. (고정)
```

> 입력이 일부만 있으면(예: 토크 하나) 계산 가능한 종류만 카드로 보여주고, 나머지는 `missing_inputs`로 "추가 입력 필요"를 안내한다(부분 진단).

## 5. 공통 상수 (LLM이 만들지 않음)

- **등급 색**: 🔴 HIGH / 🟠 MEDIUM / 🟡 LOW / ⚪ NONE / 🔵 INFO
- **공통 안전문구(고정)**: `"규칙 기반 보조 진단입니다. 정지·재가동·정비 승인은 현장 안전 책임자와 설비 담당자가 판단합니다."`
  - 매 답변 끝에 렌더러가 1회 부착 → **반복 보일러플레이트 제거**, 토큰 절약, 문구 일관성
- **failure_type 라벨**: 기존 `FAILURE_TYPE_LABELS`(OSF→"OSF(과부하 계열)" 등) 재사용
- **계산식 문구 생성기**: 고장 종류별 `formula` 문자열을 만드는 룰 엔진 함수(아래 §4.5 매핑표 기준). LLM이 아니라 코드가 생성

---

## 6. 텍스트 렌더러 (`answer_text`)

구조화 필드를 **하나의 텍스트**로 합치는 deterministic 함수. 용도:
- `output_safety_gate`의 안전성 검사 입력
- `memory_writer`가 저장하는 assistant turn
- 프론트가 구조화 렌더링을 못 하는 환경(로그/CLI)에서의 fallback

규칙: 섹션 헤더 → bullet 순으로 단순 직렬화, 공통 안전문구 부착. LLM 호출 없음.

---

## 7. 프론트 컴포넌트 매핑(참고)

```text
risk_level/headline → <RiskBanner level color>
prediction          → <VerdictBanner failure_expected verdict_label failure_types[] confidence>
risk_items[]        → <RiskGradeCard code label level rule formula drivers>   # 종류별 계산식 카드
key_findings[]      → <BulletList>
action_steps[]      → <Checklist>      (체크 상태는 프론트 로컬)
history             → <HistorySummary + <Collapsible examples>>
evidence[]          → <CitationFootnote cid title snippet source>  (접기)
cautions + 공통문구  → <CautionBox>
missing_inputs[]    → <InputPrompt>
```

## 8. 구현 단계 (노트북 변경 지점)

1. **스키마 추가**: `FinalAnswerV2`, `PredictionVerdict`, `RiskItem`, `HistoryBlock`(+ 기존 `FinalAnswer`와 호환 위해 한동안 병행) — `CHECKPOINT_SAFE_TYPES`에 신규 타입 등록
2. **룰 엔진에 계산식 추가**: `compute_partial_risks`/`FailureRisk`가 종류별 `rule`·`formula` 문자열을 함께 산출하도록 확장(§4.5 매핑표). 기존 `detail`을 사람이 읽는 계산식으로 승격
3. **프롬프트 교체**: `FINAL_ANSWER_*_PROMPT`를 "산문 작성" → "structured output 필드 추출"로 변경. LLM은 `headline`·`key_findings`·`cautions` 표현만, **진단 수치/계산식은 입력으로 받은 prediction 값 그대로 사용**
4. **`final_answer_node` 수정**: prediction 아티팩트 → `prediction`(PredictionVerdict)·`risk_items`를 **코드가 deterministic 매핑**, LLM은 headline만 채움 → 렌더러로 `answer_text` 생성
5. **렌더러 추가**: `render_answer_text(FinalAnswerV2) -> str`, `risk_level = max(risk_items.level)` 산출, `evidence`는 `evidence_bundle.citations`에서 주입
6. **공통 상수 분리**: 안전문구/등급 색/라벨 상수화, `_fallback_final_answer`도 구조화 버전으로
7. **호환 확인**: `output_safety_gate`·`memory_writer`는 `answer_text`만 보게 유지 → 기존 안전 backstop 그대로 동작
8. **검증 셀 갱신**: T04/T05/T13 등 답변 품질 체크를 구조화 필드 기준으로 보강(예: `risk_items`에 최고등급·`formula` 존재, `action_steps` 비어있지 않음)

호환성 메모:
- `output_safety_gate`가 문자열을 검사하므로, **구조화로 바꿔도 `answer_text`만 동일하게 공급하면 안전 로직은 무변경**.
- 프론트 전이 전까지는 `answer_text`가 그대로 기존 string 출력 역할을 한다 → **점진적 전환 가능**.

## 9. 기대 효과

- 결론·위험·액션이 한눈에 → 현장 사용성 ↑
- 포맷 편차 제거 → 답변 품질 회귀 테스트가 **필드 단위로 검증 가능**
- 공통 문구 상수화 → 반복 제거, 토큰·비용 절감
- 출처가 구조화 → 클릭/접기, 신뢰성 표현 개선
